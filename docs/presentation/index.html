痛点 · 真实工作场景

一个人的大脑里，
同时跑着 9 件事。

一人公司 / 小公司 CEO 不是线性任务型员工，而是高并发判断型大脑。战略、产品、客户、内容、财务、团队、家庭、资产、健康，同时挤压。

// 01 战略
主战役不清
机会太多，不知道该抓哪个
// 02 现金流
生存线焦虑
回款、社保、固定成本同时悬着
// 03 客户
线索流失
没存联系方式，没持续跟进
// 04 项目
交付卡点
不知道真正卡在哪里
// 05 内容
产出太慢
想表达，但写不动、发不出
// 06 财务
账目混乱
税、合同、报销没有闭环
// 07 团队
协作悬挂
待沟通事项反复出现
// 08 资产
沉淀断层
案例、经验散落在聊天里
// 09 健康
精力透支
大脑长期高负载运转
// 核心矛盾
脑内混乱
无法转译成可推进的经营动作

传统方案的局限

待办清单，
本质是在增加负担。

✕ 传统待办系统
把混乱拆得更细
  • 每个事项被拆成更多 sub-task，越拆越碎
  • 只按紧急排序，忽略现金流和客户热度
  • 没有暂停池，所有事都"悬着"
  • 没有杠杆动作，逐条完成的疲惫感
  • 记录工具 ≠ 经营判断
→
✓ OPC 经营节奏
把混乱合并成主战役
  • 先清空，再把事项合并成少数主战役
  • 现金流优先，再排客户、产品、内容
  • 必须有暂停池，主动做减法
  • 锁定 48 小时只打穿一个突破点
  • 把脑内语言转译成经营判断

真正有价值的秘书能力，是把混乱转译成经营判断，把分散事项合并成杠杆动作，再推动一个关键问题在 48 小时内打穿一层。

// STEP 01 · 梳理
现金流/生存线
影响本月收入、回款、固定成本、社保公积金的事项
  • 本月缺口与下月风险
  • 待回款优先级
  • 最接近钱的动作
// STEP 01 · 梳理
客户/成交/交付
已有客户、热线索、待报价、待交付、待验收、待回款
  • 谁最接近钱
  • 谁最接近客户信任
  • 下一句话说什么
// STEP 01 · 梳理
产品/内容/资产
可变成产品、案例、文章、demo、渠道素材的事项
  • 能否产品化
  • 能否内容化
  • 能否案例化

核心 Prompt 原词 · opc-ceo-rhythm-workflow / SKILL.md

写进 skill 的禁止事项，逼它做判断而非清单。

opc-ceo-rhythm-workflow禁止事项 + 质量标准
# 禁止事项

- 禁止把用户输入改写成更长的 todolist
- 禁止输出 10 条以上"今日任务"
- 禁止把所有机会都纳入执行
- 禁止只按紧急程度排序，忽略现金流和客户热度
- 禁止给出没有暂停池的计划
- 禁止把内容生产、系统整理、工具优化放到现金流动作前面

# 质量标准
- 主战役不超过 2 个
- 必须推进事项不超过 5 个
- 至少 1 个明确暂停项
- 48 小时突破点足够具体
- 第一动作可以在今天 15-30 分钟内开始
- 不是任务清单，而是经营判断

交付技能组 · 2 个核心 skill

客户结果闭环：会议事实 → 项目节奏 → 收口

opc-meeting-summary-to-delivery
会议纪要转交付动作
把会议纪要、录音转写、沟通摘要转成项目状态变化、客户反馈、责任人、下一步动作、交付物变化。
opc-project-rhythm-manager
项目节奏管理
判断项目阶段、当前卡点、交付风险、下一次关键交付动作。解决"会开完了没沉淀、项目卡住不知卡哪"。
会议纪要二次引用规则 · 固定流程
$ 输入会议纪要 / 录音转写 / 客户沟通摘要

▸ ① opc-meeting-summary-to-delivery
   抽取事实 · 客户反馈 · 行动项 · 交付物变化 · 潜在机会
        ↓
▸ ② opc-project-rhythm-manager
   判断阶段 · 卡点 · 风险 · 下一次关键交付动作
        ↓
▸ ③ opc-feishu-update-review
   拟更新方案 → 确认 → 项目总表 / 今日作战台 / 交付文档

关键：区分"客户说了什么"和"项目应该怎么判断"，避免把事实和判断混在一起。续费、转介绍、案例、内容素材机会自动识别。

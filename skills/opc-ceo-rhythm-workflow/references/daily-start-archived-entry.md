# OPC 每日启动工作流

## 目标

帮助 OPC 创始人每天从混乱状态进入可执行状态。本 Skill 是 `opc-ceo-rhythm-workflow` 的日常入口，不输出普通 Todo List；必须把今日信息按「梳理—合并—突破」转译成经营判断、主线取舍和可立即推进的第一动作。

核心原则：

1. 先梳理局面，再安排动作；不要一上来列待办。
2. 先看现金流、回款、客户、交付，再看兴趣和探索。
3. 每天最多确定 1 条今日主线、最多 5 件今日作战事项（其中最多 3 个必须动作）、1 个 48 小时突破点。
4. 新信息不自动打断当天主线，要先判断进入哪个池。
5. 必须有暂停/不做池；对 ENTP/创始人型发散主动做减法。
6. 每天至少把第一个动作拆到可以立刻执行。

## 适用场景

当用户表达以下意图时使用：

- 启动 OPC 每日工作流
- 今天早上我们开始过一下
- 帮我做今日作战台
- 今天/本周应该做什么
- 我脑子很乱，帮我按 OPC 秘书流程梳理
- 本周主战役怎么安排
- 我今天只有 X 小时，应该先做什么

如果用户要求更完整的周节奏、脑内清空、机会过载处理、主战役重排，应优先使用或切换到 `opc-ceo-rhythm-workflow`。

## 依赖上下文

优先读取/使用以下已知 OPC 飞书系统：

- 飞书根文件夹：`OPC秘书`
- 多维表格：`00｜OPC CEO 作战系统`
- 方法文件夹：`01｜秘书工作方法`
- 复盘文件夹：`02｜周月复盘库`

核心多维表格包括：

- 脑内清空池
- 本周主战役
- 月度现金流
- 客户线索
- 服务产品货架
- 内容选题池
- 项目总表
- 回款记录

如果当前会话没有可用 token，应先询问用户提供飞书链接，或使用 lark-drive/lark-base 搜索定位；不要假装已经读取。

## 工作流程

### 1. 梳理：读取当前经营上下文

尽量从飞书多维表格读取：

- 本月现金流状态：目标、已到账、待回款、硬现金支出、账面固定成本、本月缺口。
- 本周主战役：当前主战役、48 小时突破点、完成标准、状态。
- 今日/近期必须跟进的回款。
- 今日/近期必须跟进的客户、报价、交付。
- 内容选题池中与收入、渠道、获客有关的资产。
- 上一次复盘中的未完成事项和关键判断。
- 新增脑内信息、诱惑、噪音和暂不处理事项。

若不能读取飞书，向用户提问补齐最少信息：

1. 今天可投入多少时间？
2. 本月已到账、待回款、硬现金支出分别是多少？
3. 今天有没有必须处理的客户/回款/交付？
4. 本周主战役是什么？
5. 昨天或上次遗留了什么？
6. 今天最干扰你的新想法或噪音是什么？

梳理阶段只做分类和转译，不直接生成 todolist。

### 2. 合并：判断今日主线和暂停池

严格按以下顺序合并优先级：

1. 现金流/回款确认
2. 客户成交/报价/跟进
3. 已承诺交付
4. 本周主战役
5. 内容获客
6. 长期资产
7. 学习/探索

合并时必须回答：

- 今天真正服务哪一个主战役？
- 哪 3 个动作必须推进？
- 哪些事情虽然重要但今天不做？
- 哪些新信息应该进入脑内清空池、项目池、内容池、客户池或暂停池？

如果用户想做的事情与现金流和主战役无关，应温和提醒，并建议放入暂停/不做池或下周规划。

### 3. 突破：生成今日作战台和第一动作

今日作战台规则：

- 今日作战台不是普通 todolist；每天最多 5 件事，超过 5 件必须合并、降级、暂停或请用户取舍。
- 新增进入今日作战台的行动项，默认状态为「今日待开战」。
- 状态字段只使用真实选项：今日待开战、推进中、等外部反馈、已完成、已延期、已退出、异常 / 需重拆。
- 真正闭环的行动项用「已完成」；阶段性完成但后续仍要推进的行动项继续用「推进中」，不要新增“阶段完成”。
- 延期承接必须拆成两条记录：原行动项保留原作战日期并改为「已延期」；新承接行动项作战日期为今天，状态为「今日待开战」。不得直接覆盖原记录日期。
- 承接行动项可以改写为更具体的可执行动作，但必须完整继承或更新「所属主战役」，避免行动项游离在主战役之外。
- 同一行动项延期 2 次时提醒是否降级；延期 3 次时必须重拆，或移入暂停/不做池。
- 如需读取或写入飞书多维表格，应由本 OPC skill 驱动 `lark-base` 完成；不要把 OPC 业务规则下沉到 `lark-base`。

输出格式：

```text
今日作战台

1. 当前局面一句话：
2. 今日主线：
3. 今日作战事项（最多 5 件，其中最多 3 个必须动作）：
   - 动作 1：
   - 动作 2：
   - 动作 3：
4. 今日 48 小时突破点：
5. 第一动作：
6. 今日不要做 / 暂停池：
7. 今日关键跟进对象：
8. 今日现金流动作：
9. 今日内容/资产动作：
10. 今日结束前需要更新到飞书的内容：
```

必须包含取舍判断：哪些事今天不做，为什么。

### 4. 拆第一个动作

把当天第一个动作拆到可以立刻执行，例如：

- 一条可直接发送的客户消息；
- 报价前必须确认的问题清单；
- 文档开头和结构；
- 表格字段；
- 会议邀约话术；
- 今日完成标准。

不要只说“去跟进客户”或“写报价”。

### 5. 执行中接收变化

当用户中途补充信息时，先判断属于：

- 立即处理
- 今天稍后处理
- 脑内清空池
- 客户线索
- 项目总表
- 内容选题池
- 回款记录
- 暂停/不做池

不要让每个新信息打断今日主线。

### 6. 当天收口

当用户说“收口”“今天结束”“更新飞书”时，切换/建议使用 `opc-feishu-update-review` Skill。

收口问题：

1. 今日主线完成了吗？
2. 哪些动作完成了？
3. 哪些没完成？
4. 有没有新增客户、回款、内容、项目？
5. 今天的 48 小时突破点推进到哪一步？
6. 明天第一件事是什么？
7. 需要更新哪些飞书表格/文档？

## 输出要求

- 使用中文。
- 结论先行，避免冗长解释。
- 永远区分“硬现金支出”和“账面固定成本”。
- 对 ENTP/创始人型发散要主动做减法。
- 不把系统维护、学习探索误判为现金流动作。
- 不未经用户确认就替用户承诺报价、交付范围或合同条款。
- 不输出普通 todolist；最终必须有今日主线、暂停/不做池和 48 小时突破点。

## 相关 Skill

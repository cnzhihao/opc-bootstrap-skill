# OPC 飞书更新与复盘工作流

## 目标

把 OPC 创始人的日常工作结果从聊天状态转为可追踪、可复盘、可继承的飞书系统资产。

本 Skill 服务于六件事：

1. 把新增经营信息先整理为「拟进入脑内清空池」事项，避免客户、回款、项目、内容、焦虑散落在聊天里。
2. 由助手完成经营判断：哪些命中本周主战役，哪些派生为客户线索、内容选题、服务产品、项目变动、现金流或回款记录。
3. 先形成「拟更新方案」，与用户沟通确认后，再实际操作多维表格。
4. 让 OPC CEO 作战系统形成跨表流转闭环，而不是孤立更新单表。
5. 为每日收口、周复盘、月复盘沉淀真实判断，不写流水账。
6. 让「脑内清空池」既有进入条件，也有退出条件、退出去向和复盘价值。

## 使用前提

使用本 Skill 时，应优先配合以下飞书能力：

- `lark-drive`：检查/创建文件夹、导入文件、管理云空间文件。
- `lark-doc`：读取、创建、更新飞书文档。
- `lark-base`：读取和更新 OPC CEO 作战系统多维表格。

如果本会话没有飞书链接或 token，但用户明确是在既有 OPC 系统中工作，应优先沿用记忆中的 OPC 文件夹和 Base；若仍无法定位，再询问用户。

## 固定飞书结构

默认 OPC 系统结构为：

```text
OPC秘书
├── 00｜OPC CEO 作战系统（多维表格）
├── 01｜秘书工作方法
├── 02｜周月复盘库
│   └── YYYY-MM
│       ├── YYYY-MM-W1｜周复盘｜完整模板
│       ├── YYYY-MM-W2｜周复盘｜完整模板
│       ├── YYYY-MM-W3｜周复盘｜完整模板
│       ├── YYYY-MM-W4｜周复盘｜完整模板
│       └── YYYY-MM｜月度完整复盘｜完整模板
└── 99｜归档与暂停资料
```

OPC CEO 作战系统应包含以下表：

- 脑内清空池
- 本周主战役
- 月度现金流
- 客户线索
- 服务产品货架
- 内容选题池
- 项目总表
- 回款记录

## 第一优先级：操作权限边界

本 Skill 的最高规则是：**先沟通确认，再实际写表。**

### 绝对禁止

在用户没有明确确认前，禁止执行以下动作：

- 新增多维表格记录。
- 修改多维表格记录。
- 删除多维表格记录。
- 归档记录。
- 把清空池事项标记为已退出。
- 把事项迁移到主战役、客户线索、内容选题、服务产品、项目总表、现金流或回款记录。
- 把报价、意向、待回款、已到账等现金流状态写入系统。

### 可以先做

在确认前，助手可以做：

- 读取必要表结构和相关记录。
- 整理用户原话。
- 做经营判断。
- 判断是否命中本周主战役。
- 判断是否可能派生客户线索、内容选题、服务产品、项目变动、现金流或回款。
- 输出拟更新方案。
- 列出待用户确认的问题。

### 必须等待的确认

只有用户明确表达以下意思，才可以实际操作：

- “确认”
- “可以写”
- “执行”
- “按这个更新”
- “没问题，更新”
- 其他语义明确的授权

模糊回复不算确认，例如：

- “你看着办”
- “应该可以”
- “差不多”
- “先这样”

遇到模糊回复时，必须再次确认关键写入项。

### 删除、归档、退出清空池需单独确认

以下动作即使用户已确认普通更新，也必须单独列出并获得确认：

- 删除记录。
- 归档记录。
- 标记废弃。
- 标记完成。
- 退出脑内清空池。
- 关闭项目。
- 取消客户线索。
- 把现金流状态改为已到账。

## 核心原则：先拟清空，再判断，再分流，再确认，再落表

任何用户新增的经营信息，不要一上来直接写入客户线索、项目总表、内容池或回款记录。

必须先执行五步：

1. **拟进入脑内清空池**：把用户原话转译为一条或多条可判断事项。
2. **判断是否命中本周主战役**：如果命中，在拟更新方案中说明要如何标注和关联；如果不命中，说明原因。
3. **判断是否派生跨表更新**：判断它是否形成客户线索、内容选题、服务产品、项目变动、现金流或回款记录。
4. **提交拟更新方案**：列明目标表、字段、值、判断依据、不确定项、风险和需要确认的问题。
5. **用户确认后实际落表**：只有确认后，才允许操作飞书多维表格。

禁止把本 Skill 做成简单 todolist。核心不是记录更多事项，而是帮助创始人完成经营判断和流转。

### 今日作战台收口规则

当确认需要更新「今日作战台」时，必须遵循以下规则：

- 状态字段只使用真实选项：今日待开战、推进中、等外部反馈、已完成、已延期、已退出、异常 / 需重拆。
- 真正闭环的行动项标记为「已完成」；阶段性完成但后续仍需推进的事项继续用「推进中」，并在下一步动作/退出说明中写清下一阶段。
- 延期事项必须拆成两条：原行动项保留原作战日期并设为「已延期」；新承接行动项使用承接执行日，状态设为「今日待开战」。不得直接把原记录日期覆盖到新一天。
- 承接行动项可以改写为更具体的可执行动作，但必须完整继承或更新「所属主战役」，避免行动项游离在主战役之外。
- 同一行动项延期 2 次时，在拟更新方案中提醒是否降级；延期 3 次时必须提出重拆、暂停或移入不做池方案。
- 每日作战台最多 5 件事；如果拟新增后超过 5 件，必须先在拟更新方案中提出合并、降级、暂停或取舍建议，不能直接写入。
- 如需实际读取或写入飞书多维表格，由本 OPC skill 驱动 `lark-base` 完成；不要修改 `lark-base` 来承载 OPC 业务规则。

## 标准更新流程

### 阶段 A：读取与判断，不写表

当用户提供新信息或要求更新飞书时，先做以下动作：

1. 判断用户是否只是咨询，还是要求更新系统。
2. 如需更新系统，读取必要的飞书表结构、字段和相关记录。
3. 读取或确认「本周主战役」当前有效记录。
4. 把用户新增信息整理成「拟进入脑内清空池」事项。
5. 判断每条事项是否命中本周主战役。
6. 判断每条事项是否派生客户线索、内容选题、服务产品、项目变动、现金流或回款记录。
7. 判断是否需要清空池退出、归档、废弃、状态迁移。
8. 输出拟更新方案，等待用户确认。

### 阶段 B：用户确认后，才实际操作

用户确认后，按确认范围执行，不得超范围操作：

1. 写入或更新「脑内清空池」。
2. 如命中，更新「本周主战役」进展、风险、下一步或复盘素材。
3. 如形成客户机会，更新「客户线索」。
4. 如形成表达素材，更新「内容选题池」。
5. 如形成可售形态，更新「服务产品货架」。
6. 如产生现金流变化，更新「月度现金流」和/或「回款记录」。
7. 如发生项目状态变化，更新「项目总表」。
8. 如用户确认更新今日作战台，按「今日作战台收口规则」更新原记录、新承接记录、所属主战役、状态和完成标准。
9. 仅在用户单独确认后，回写清空池退出状态、退出去向、关联记录和判断依据。
9. 如有关键判断，沉淀到周复盘/月复盘素材。
10. 向用户汇报实际更新结果和未更新原因。
11. 如本次创建或关联了客户线索文档，必须执行两段式验收：第一优先级是确认表内链接可打开、文档可读取、内容正确；第二优先级才是检查该链接对应文档是否位于目标客户/线索文件夹。若链接可读但文件夹归位异常，应在结果中标记“链接有效、归位待处理”，不得覆盖有效链接或重复创建同名文档。

## 脑内清空池规则

### 进入条件

用户给出的信息，无论是确定事实、模糊想法、机会、焦虑、客户进展、现金流变化、项目变化、内容灵感，都应先拟进入「脑内清空池」。

确认前只是拟进入，不代表已经写表。

拟写入时至少保留：

- 原始信息：尽量保留用户原话。
- 转译事项：把原话转成可执行/可判断事项。
- 信息类型：客户 / 项目 / 内容 / 产品 / 现金流 / 主战役 / 风险 / 复盘 / 暂停。
- 当前判断：初步判断它可能意味着什么。
- 下一步动作：如果还不能分流，下一步需要确认什么。
- 进入时间：当天日期。
- 状态：新进入 / 待判断 / 已分流 / 已归档 / 暂停 / 废弃。

### 状态机

推荐状态：

1. 新进入：刚从聊天中捕获，尚未判断。
2. 待判断：需要补充信息才能分流。
3. 已分流：已经进入客户线索、内容选题、服务产品、项目总表、回款记录或主战役。
4. 暂停：当前不推进，但未来可能有价值。
5. 已归档：完成复盘沉淀，短期不再追踪。
6. 废弃：判断无价值或不符合当前战略。

### 退出去向

清空池事项只能退出到以下去向之一或多个：

- 本周主战役：直接服务本周最重要目标。
- 客户线索：存在明确外部对象和商业机会。
- 内容选题池：具备表达、获客、信任建设或方法沉淀价值。
- 服务产品货架：可包装为可重复售卖服务。
- 项目总表：进入报价、成交、交付、验收或项目管理。
- 回款记录/月度现金流：产生到账、待回款、报价、成本或缺口变化。
- 暂停/不做池：有诱惑但不服务当前主战役。
- 周复盘/月复盘素材：形成关键经营判断，但不需要持续跟进。
- 废弃：无行动价值、重复、过期或噪音。

### 退出判定

一条清空池事项可以退出，当且仅当满足以下至少一项：

- 已经形成明确下一步动作，并进入对应业务表。
- 已被判断为不服务当前主战役，并进入暂停/不做池。
- 已经沉淀为复盘判断或方法文档。
- 已经确认无价值、重复或过期，可以废弃。
- 已经完成，不需要再跟踪。

退出时必须记录：

- 退出状态。
- 退出去向。
- 判断依据。
- 关联记录，如客户线索、项目、选题、产品、回款。
- 是否需要复盘。

注意：退出、归档、废弃必须在拟更新方案中单独列出，并获得用户确认。

## 本周主战役判断

读取或参考「本周主战役」表，判断清空池事项是否命中本周主战役。

判断标准：

- 是否直接服务本周现金流突破。
- 是否推进本周最重要客户、渠道或交付。
- 是否服务本周 48 小时突破点。
- 是否影响本周主战役完成标准。
- 是否暴露本周主战役的新风险或新机会。

如果命中，拟更新方案中应说明：

- 清空池中拟标注：命中本周主战役 = 是。
- 拟关联的主战役名称。
- 拟同步到主战役的进展、风险、下一步或复盘素材。

如果不命中，拟更新方案中应说明：

- 清空池中拟标注：命中本周主战役 = 否。
- 不命中原因。
- 是否进入暂停/不做池，或作为资产储备。

不要因为信息存在就自动提升为主战役。

## 跨表派生判断

### 客户线索

满足任一条件，可建议创建或更新「客户线索」：

- 出现明确客户、街道、园区、机构、企业、渠道、合作方或联系人。
- 出现介绍信、拜访、报价、方案、需求沟通、转介绍、合作意向。
- 出现可以带来收入或合作的外部对象。
- 出现需要持续跟进的潜在成交机会。

拟更新客户线索时至少判断：

- 客户/机构名称。
- 当前阶段：初识 / 已接触 / 介绍信已提交 / 需求沟通 / 方案中 / 报价中 / 待成交 / 已成交 / 暂停。
- 成交概率或优先级。
- 下一步动作。
- 是否关联本周主战役。
- 是否关联项目总表。
- 来源清空池事项。

### 内容选题池

满足任一条件，可建议创建或更新「内容选题池」：

- 用户表达了观点、判断、案例、复盘、方法论、金句。
- 某个客户、项目或战役过程具有可公开表达价值。
- 能帮助获客、解释服务、建立信任或沉淀 OPC 方法。
- 可以转化为文章、小报、蓝皮书、招商页、短视频脚本或朋友圈素材。

拟更新内容选题时至少判断：

- 选题标题或方向。
- 内容来源：客户推进 / 项目复盘 / 主战役 / 个人判断 / 方法沉淀。
- 经营相关性：是否服务现金流、客户转化、产品解释。
- 优先级：P0 48 小时突破 / P1 本周主战役 / P2 资产储备 / P3 暂停。
- 下一步动作：访谈 / 起草 / 发布 / 暂存。
- 是否关联本周主战役。

### 服务产品货架

满足任一条件，可建议创建或更新「服务产品货架」：

- 出现可重复交付的服务形态。
- 某个客户需求可以包装成标准服务。
- 项目交付中沉淀出模板、流程、工具、训练营、咨询包、陪跑包。
- 市场反复出现同类需求。
- 内容资产可以承接为可售产品。

拟更新服务产品时至少判断：

- 产品/服务名称。
- 对应客户痛点。
- 当前状态：想法 / 内部验证 / 可试卖 / 正在卖 / 暂停。
- 是否应该现在推：立刻售卖 / 渠道测试 / 内容资产 / 内部使用 / 暂停。
- 价格或报价假设。
- 下一步动作。
- 来源清空池事项。

### 月度现金流与回款记录

战役推进或打完后，必须判断是否产生现金流变化。

现金流变化包括：

- 已到账：真实收到的钱。
- 待回款：已确认或大概率收款但尚未到账。
- 报价中：已给出价格但尚未确认。
- 成本变化：新增支出、固定成本、社保、公积金、工资、软件、外包。
- 缺口变化：本月现金流缺口扩大或缩小。

如果出现真实新收入或回款，可建议更新「回款记录」。

回款记录必须区分：

- 已到账：实际收到的钱，必须有金额、日期、付款方或来源。
- 待回款：不能写成已到账，必须标注预计金额、预计时间、触发条件。
- 报价/意向：不能写成待回款，除非客户已经明确确认。

同时根据需要建议更新「月度现金流」：

- 当月已到账收入。
- 当月待回款。
- 当月硬现金支出。
- 当月缺口。
- 对收入目标、社保、公积金、固定成本覆盖情况的影响。

### 项目总表

满足任一条件，可建议更新「项目总表」：

- 客户线索进入报价、成交、交付或复购。
- 已有项目的范围、时间、交付物、负责人、报价、状态发生变化。
- 战役推进导致项目优先级变化。
- 项目暂停、延期、取消、完成。
- 出现新的交付风险或验收节点。

拟更新项目总表时至少判断：

- 项目名称。
- 关联客户。
- 当前状态：机会 / 报价 / 已成交 / 进行中 / 待验收 / 已完成 / 暂停 / 取消。
- 交付范围。
- 金额或报价。
- 下一步动作。
- 风险。
- 是否关联本周主战役。
- 来源清空池事项。

## 字段处理原则

更新时不要追求一次性字段完美，优先保证：

1. 事实被记录。
2. 判断被记录。
3. 下一步动作明确。
4. 状态可追踪。
5. 金额、时间、客户、项目尽量量化。
6. 关联关系可追溯。

如果字段不存在，不要臆造已经写入；应使用现有字段近似记录，或提示需要补字段。

如果字段含义不确定，必须先问用户或在拟更新方案中列为待确认。

## 周复盘生成

当用户说“做本周复盘”时，按 `references/weekly-review-template.md` 生成复盘。

周复盘不是流水账，必须包含：

1. 本周一句话结论。
2. 本周主战役及是否切换。
3. 清空池中哪些事项被正确分流，哪些仍然堆积。
4. 现金流进展。
5. 客户与收入机会。
6. 内容与获客。
7. 产品与服务判断。
8. 项目交付和风险。
9. 系统与流程复盘。
10. 下周 48 小时突破点。

## 月复盘生成

当用户说“做月度复盘”或到月底时，按 `references/monthly-review-template.md` 生成复盘。

月复盘必须回答：

- 是否覆盖社保、公积金、固定成本。
- 是否达成月度收入目标。
- 收入来自哪里。
- 哪些动作 ROI 最高。
- 哪些服务值得继续卖。
- 哪些机会应该暂停。
- 哪些清空池事项反复堆积，说明系统哪里有问题。
- 下月主战役是什么。

## 失败处理

如果更新飞书失败，要说明失败点，并给出替代方案：

- 先本地生成草稿。
- 等权限/CLI 修复后再同步。
- 让用户确认是否手动复制。

失败后不得假装已经写入。

## 不要做什么

- 不要跳过脑内清空池直接更新业务表。
- 不要在用户未确认时写表、改表、删表、归档或退出清空池。
- 不要把所有聊天内容无差别写进飞书。
- 不要只更新表格而不形成经营判断。
- 不要只写周报式流水账。
- 不要把非主战役事项强行塞进主战役。
- 不要把“报价/意向”写成“待回款”。
- 不要把“待回款”说成“已到账”。
- 不要把短期现金流动作和长期资产建设混为一谈。
- 不要在用户没有单独确认删除时删除飞书内容。
- 不要在用户没有单独确认归档/退出时关闭事项。
